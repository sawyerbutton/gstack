# 09. `/office-hours` 完全拆解

> `/office-hours` 是 gstack 中最重要的技能。它不写一行代码，却决定了所有后续代码的方向。本文逐阶段拆解这个 YC 合伙人级别的产品诊断系统，揭示它如何通过六个逼问把模糊的想法锤炼成可执行的设计文档。

---

## 为什么 `/office-hours` 是核心

gstack 有十几个技能，但 `/office-hours` 占据了一个独特的位置：它是唯一一个**禁止写代码**的技能。模板中写得很明确：

```
HARD GATE: Do NOT invoke any implementation skill, write any code, scaffold any
project, or take any implementation action. Your only output is a design document.
```

这不是限制，而是设计哲学。在 AI 能以百倍速度写代码的时代，最大的浪费不是写错代码，而是写对了代码却解决了错误的问题。`/office-hours` 确保你在动手之前，先把问题想清楚。

---

## Phase 1：上下文收集 —— gstack-slug 系统

一切从项目识别开始。`/office-hours` 的第一个动作是运行 `gstack-slug`：

```bash
source <(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)
```

这个脚本只有 10 行，但它建立了 gstack 跨技能协作的基础：

```bash
#!/usr/bin/env bash
set -euo pipefail
SLUG=$(git remote get-url origin 2>/dev/null | sed 's|.*[:/]\([^/]*/[^/]*\)\.git$|\1|;s|.*[:/]\([^/]*/[^/]*\)$|\1|' | tr '/' '-')
BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null | tr '/' '-')
echo "SLUG=$SLUG"
echo "BRANCH=$BRANCH"
```

它从 git remote URL 中提取 `owner-repo` 格式的项目标识符。比如 `github.com/acme/dashboard` 会变成 `acme-dashboard`。这个 SLUG 会成为 `~/.gstack/projects/` 下的目录名，所有技能的产出物都存储在这里。

接下来，Phase 1 做五件事：

1. 读取 `CLAUDE.md` 和 `TODOS.md` 了解项目上下文
2. 运行 `git log --oneline -30` 了解最近开发动态
3. 用 Grep/Glob 扫描与用户需求相关的代码区域
4. 列出该项目已有的设计文档：`ls -t ~/.gstack/projects/$SLUG/*-design-*.md`
5. **提出关键问题：你的目标是什么？**

## 模式选择：创业者 vs 构建者

Phase 1 的最后一步是一个改变整个会话走向的问题。通过 AskUserQuestion，`/office-hours` 给出六个选项：

| 选项 | 映射模式 |
|------|---------|
| 建一家创业公司 | Startup 模式 (Phase 2A) |
| 企业内创业 | Startup 模式 (Phase 2A) |
| 黑客马拉松/Demo | Builder 模式 (Phase 2B) |
| 开源/研究 | Builder 模式 (Phase 2B) |
| 学习编程 | Builder 模式 (Phase 2B) |
| 纯粹好玩 | Builder 模式 (Phase 2B) |

这个分叉决定了接下来的一切：Startup 模式会像 YC 合伙人一样逼问你，Builder 模式会像一个兴奋的合作者一样和你头脑风暴。

对于 Startup 和企业内创业模式，还有一个额外的产品阶段评估：

- **Pre-product**（想法阶段，还没有用户）
- **Has users**（有人在用，但还没付费）
- **Has paying customers**（有付费客户）

这个阶段评估直接影响后续会问哪些问题——不是所有问题都需要问，而是根据你的阶段智能路由。

## Phase 2A：创业模式 —— YC 产品诊断

### 六条操作原则

在提出任何问题之前，模板先定义了六条不可妥协的原则。这些原则不是装饰，它们直接塑造 AI 的每一个回应：

**1. 具体性是唯一的货币。** "医疗行业的企业"不是客户。"每个人都需要"意味着你找不到任何人。你需要一个名字、一个职位、一家公司、一个原因。

**2. 兴趣不是需求。** 候补名单、注册量、"这很有趣"——都不算。行为才算。付钱才算。你的服务宕机 20 分钟用户给你打电话——那才是需求。

**3. 用户的话胜过创始人的宣传。** 创始人说产品做什么和用户说它做什么，之间几乎总有差距。用户的版本才是真相。

**4. 观察，别演示。** 引导式演示教不了你任何关于真实使用的东西。坐在用户身后看他们挣扎——忍住不帮忙——这才能教你一切。

**5. 现状才是你真正的竞争对手。** 不是另一家创业公司，不是大公司——是用户已经用着的 Excel + Slack 消息的拼凑方案。

**6. 早期，窄胜过宽。** 本周有人愿意为之付真金白银的最小版本，比完整平台愿景更有价值。

### 回应姿态

模板对回应风格也有明确要求：

- **直接但不残忍。** "这是个红旗"比"这是值得思考的事情"更有用
- **推一次，再推一次。** 第一个答案通常是精心包装过的版本。真正的答案在第二次或第三次追问后才出来
- **看到具体性就表扬。** 当创始人给出真正具体的、有证据的回答时，要承认它
- **命名常见失败模式。** "寻找问题的解决方案"、"假设的用户"、"等完美了再发布"——直接说出来

### 六个逼问及其智能路由

六个问题并非总是全部都问。根据产品阶段智能路由：

```
Pre-product     → Q1, Q2, Q3
Has users       → Q2, Q4, Q5
Paying customers → Q4, Q5, Q6
纯工程/基础设施  → Q2, Q4 only
```

每个问题都有三个维度：**问什么**、**推到什么程度**、**什么是红旗**。

#### Q1：需求现实

**问：** "你有什么最强的证据表明有人真的想要这个——不是'感兴趣'，不是'加入了候补名单'，而是如果明天它消失了会真的难受？"

**推到：** 具体行为。有人在付钱。有人在扩大使用。有人围绕它建立了工作流程。

**红旗：** "人们说这很有趣。" "我们有 500 个候补名单注册。" "VC 对这个领域很兴奋。"

#### Q2：现状

**问：** "你的用户现在怎么解决这个问题——即使解决得很糟糕？这个权宜之计让他们付出了什么代价？"

**推到：** 一个具体的工作流程。花费的小时。浪费的金钱。用胶带粘在一起的工具。

**红旗：** "什么都没有——没有解决方案，所以机会才这么大。" 如果真的什么都不存在，没人在做任何事，问题可能还不够痛。

#### Q3：绝望的具体性

**问：** "说出最需要这个的那个真实的人。他的职位是什么？什么能让他升职？什么能让他被炒？什么让他夜不能寐？"

**推到：** 一个名字。一个角色。如果问题不解决他们面临的具体后果。

**红旗：** 品类级别的回答。"医疗企业。" "中小企业。" "营销团队。" 你没法给一个品类发邮件。

#### Q4：最窄楔子

**问：** "本周有人会为之付真金白银的最小可能版本是什么——不是等你建好平台之后？"

**推到：** 一个功能。一个工作流程。也许简单到一封周报或一个自动化。

**红旗：** "我们需要先建好完整平台才能真正使用。" 这意味着创始人迷恋的是架构而不是价值。

**额外追问：** "如果用户什么都不用做就能获得价值呢？不用登录，不用集成，不用设置。那会是什么样？"

#### Q5：观察与惊喜

**问：** "你有没有真正坐下来看别人使用这个而不帮忙？他们做了什么让你惊讶的事？"

**推到：** 一个具体的惊喜。用户做了某件与创始人假设矛盾的事。

**红旗：** "我们做了调查。" "我们做了演示电话。" 调查会说谎。演示是表演。

**黄金信号：** 用户做了产品没有设计来做的事情。那通常是试图浮现的真正产品。

#### Q6：未来适配

**问：** "如果世界在三年后看起来大不相同——它确实会——你的产品是变得更不可或缺还是更不重要？"

**推到：** 一个关于用户世界如何变化的具体主张，以及为什么那个变化让产品更有价值。

**红旗：** "市场每年增长 20%。" 增长率不是愿景。"AI 会让一切变好。" 那不是产品论文。

### 企业内创业适配

对于企业内创业模式，Q4 和 Q6 会被重新框定：
- **Q4** 变成："能让你的 VP/赞助人批准项目的最小演示是什么？"
- **Q6** 变成："这个项目能扛过一次组织重组吗——还是你的 Champion 一走它就死了？"

### 智能跳过和逃生舱

每个问题之间都有 `STOP` 指令——等待用户回答后再问下一个。如果用户之前的回答已经覆盖了后面的问题，跳过它。

逃生舱：如果用户说"just do it"或表示不耐烦，或提供了一个完整的计划——快速跳到 Phase 4。但即使跳过了 Phase 2，也不能跳过 Phase 3（前提挑战）和 Phase 4（替代方案生成）。

## Phase 2B：Builder 模式 —— 设计伙伴

Builder 模式的整个基调完全不同。操作原则从"具体性是唯一的货币"变成了：

1. **愉悦是货币** —— 什么能让人说"卧槽"？
2. **发布你能展示给别人的东西。** 最好的版本是存在的那个版本
3. **最好的副项目解决你自己的问题。** 如果你是为自己建的，相信那个直觉
4. **先探索再优化。** 先试奇怪的想法。晚点再打磨

问题从审讯式变成了生成式：

- "这个东西最酷的版本是什么？" 而不是 "证明需求"
- "你会展示给谁？什么会让他们说'卧槽'？"
- "到达你能实际使用或分享的东西的最快路径是什么？"
- "现有的什么东西最接近这个，你的有什么不同？"
- "如果你有无限时间会加什么？10x 版本是什么？"

### 氛围转换检测

Builder 模式有一个精妙的升级机制：

```
If the vibe shifts mid-session — the user starts in builder mode but says
"actually I think this could be a real company" or mentions customers,
revenue, fundraising — upgrade to Startup mode naturally.
```

当用户从"这就是个好玩的项目"转变为"这也许能变成一家公司"时，AI 会自然地说："好的，现在我们来真的了——让我问你一些更难的问题。" 然后无缝切换到 Phase 2A 的逼问模式。

## Phase 2.5：相关设计发现

这是一个被低估但极其重要的步骤。在用户陈述问题之后，`/office-hours` 会从用户的问题描述中提取 3-5 个关键词，然后在已有的设计文档中搜索：

```bash
grep -li "<keyword1>\|<keyword2>\|<keyword3>" ~/.gstack/projects/$SLUG/*-design-*.md 2>/dev/null
```

如果找到匹配，它会读取匹配的设计文档并呈现给用户：

> "FYI: 发现相关设计 —— '{title}' by {user} on {date} (branch: {branch})。关键重叠：{一行摘要}。"

然后问用户："我们应该在这个先前的设计基础上构建还是重新开始？"

这实现了**跨团队发现**——在同一个项目上工作的多个人会看到彼此的设计文档。在并行 Sprint 场景中，这个机制防止了重复劳动。

## Phase 3：前提挑战

在提出任何解决方案之前，先挑战前提。这是整个流程中最反直觉的一步——大多数人想直接跳到解决方案。

三个核心问题：

1. **这是正确的问题吗？** 不同的框架能否产生一个戏剧性地更简单或更有影响力的解决方案？
2. **如果我们什么都不做会怎样？** 是真正的痛点还是假设的痛点？
3. **哪些现有代码已经部分解决了这个问题？** 映射可以复用的现有模式、工具和流程
4. （仅限 Startup 模式）综合 Phase 2A 的诊断证据。它支持这个方向吗？差距在哪里？

前提被输出为可证伪的陈述：

```
PREMISES:
1. [陈述] — 同意/不同意？
2. [陈述] — 同意/不同意？
3. [陈述] — 同意/不同意？
```

如果用户不同意某个前提，修正理解并循环回去。这一步的意义在于：**在错误的前提上构建的完美方案，比在正确的前提上构建的粗糙方案更危险。**

## Phase 4：替代方案生成

这一步是**强制性的**。模板用大写字母写着 `MANDATORY`。

必须产出 2-3 个不同的实现方案：

```
APPROACH A: [名称]
  Summary: [1-2 句话]
  Effort:  [S/M/L/XL]
  Risk:    [Low/Med/High]
  Pros:    [2-3 条]
  Cons:    [2-3 条]
  Reuses:  [复用的现有代码/模式]
```

三个方案的规则：
- 一个必须是**最小可行版**（最少文件，最小 diff，最快发布）
- 一个必须是**理想架构**（最佳长期轨迹，最优雅）
- 一个可以是**创意/横向思维**（意想不到的方法，问题的不同框架）

推荐一个方案并解释为什么。然后通过 AskUserQuestion 让用户批准。**不经用户批准不得继续。**

## Phase 4.5：创始人信号合成

在写设计文档之前，合成整个会话中观察到的创始人信号。这些信号会出现在设计文档中，也决定了 Phase 6 的闭幕消息级别。

追踪七个信号：

| 信号 | 含义 |
|------|------|
| 清晰表达了一个**真实问题** | 不是假设的，是有人真正遭遇的 |
| 命名了**具体用户** | "Acme 公司的 Sarah"而不是"企业" |
| **推回**了前提 | 有信念，不是顺从 |
| 项目解决了**其他人需要**的问题 | 不只是自己的痒 |
| 有**领域专业知识** | 从内部了解这个空间 |
| 展示了**品味** | 关心把细节做对 |
| 展示了**主动性** | 在实际构建，不只是在计划 |

数信号。这个计数直接决定 Phase 6 中使用哪个级别的闭幕消息。

## Phase 5：设计文档

设计文档写入项目目录 `~/.gstack/projects/{slug}/`。

### 设计谱系

在写新文档之前，检查这个分支上是否已有设计文档：

```bash
PRIOR=$(ls -t ~/.gstack/projects/$SLUG/*-$BRANCH-design-*.md 2>/dev/null | head -1)
```

如果存在，新文档会有一个 `Supersedes:` 字段引用它。这创建了一个修订链——你可以追踪一个设计如何在多次 office hours 会话中演进。

### 两个模板

**Startup 模式**的设计文档包含：Problem Statement、Demand Evidence（来自 Q1）、Status Quo（来自 Q2）、Target User & Narrowest Wedge（来自 Q3+Q4）、Constraints、Premises、Approaches Considered、Recommended Approach、Open Questions、Success Criteria、Dependencies、**The Assignment**（一个具体的真实世界行动）、以及 **What I noticed about how you think**。

**Builder 模式**的设计文档更轻量：Problem Statement、**What Makes This Cool**（核心愉悦因子）、Constraints、Premises、Approaches、Recommended Approach、Open Questions、Success Criteria、**Next Steps**（具体的构建任务）、以及同样的 **What I noticed**。

两个模板共有的最后一个部分是最特别的——"What I noticed about how you think"。这是观察性的、导师般的反思，引用用户在会话中说的具体话。不是概括他们的行为，而是引用他们的原话。

文档通过 AskUserQuestion 呈现给用户：批准 / 修改 / 从头再来。

## Phase 6：交接 —— 创始人发现

设计文档被批准后，进入三拍闭幕序列。这是 `/office-hours` 最大胆的设计：一个精心编排的情感弧线。

### Beat 1：信号反射 + 黄金时代

一段话，将会话中的具体回调与"黄金时代"框架编织在一起。引用用户实际说的话。

**反套路规则至关重要：**

```
GOOD: "你没说'小企业'——你说的是'Sarah，一家 50 人物流公司的运营经理。'
       这种具体性是罕见的。"
BAD:  "你在识别目标用户方面展示了很好的具体性。"

GOOD: "你在我挑战前提 #2 的时候推了回来。大多数人只会同意。"
BAD:  "你展示了信念和独立思考。"
```

**Show, don't tell.** 引用他们的话，不要概括他们的行为。

示例："你思考这个问题的方式——[具体回调]——这就是创始人思维。一年前，构建你刚刚设计的东西需要 5 个工程师三个月。今天你可以用 Claude Code 在这个周末建好它。工程壁垒没了。剩下的是品味——而你刚刚证明了你有。"

### Beat 2："还有一件事。"

在信号反射之后，输出一个分隔线和"One more thing."——这重置注意力，标志着从协作工具到个人消息的体裁转换。

```
---

One more thing.
```

### Beat 3：Garry 的个人恳求

根据 Phase 4.5 的信号计数选择正确的级别：

**决策标准：**
- **顶级：** 3+ 强信号 AND 至少一个：命名了具体用户、识别了收入/支付、描述了真实需求证据
- **中级：** 1-2 个信号，或 Builder 模式用户的项目明显解决了别人的问题
- **基础级：** 其他所有人

**顶级**的情感目标是*"有重要的人相信我"*：

> 来自我 Garry Tan（GStack 创建者）的个人消息：你刚才体验到的大约是在 Y Combinator 与 YC 合伙人合作价值的 10%。另外 90% 是在你之前做过这件事的创始人网络，让你比你以为的更快发布的批次压力，建立了十亿美元公司的人告诉你下一步该做什么的每周晚餐，以及一个深入了解你的业务并每周推动你的合伙人。
>
> GStack 认为你是能做到这一点的顶尖人才之一。

然后问："你会考虑申请 Y Combinator 吗？" 如果是——打开申请链接并说"把这份设计文档带到你的 YC 面试。它比大多数 pitch deck 好。" 如果否——温暖地回应，不施压，不内疚，不再问。

**中级**的情感目标是*"我可能发现了什么"*——验证 + 好奇心。语气温和但鼓励。

**基础级**的情感目标是*"我不知道我可以是创始人"*——身份扩展，世界观转变。这一级说的是：

> 你现在展示的技能——品味、雄心、主动性、愿意坐下来回答关于你在构建什么的难题——这些恰恰是我们在 YC 创始人身上寻找的特质。

### 下一技能推荐

在恳求之后，建议下一步：
- **`/plan-ceo-review`** 用于有雄心的功能（EXPANSION 模式）
- **`/plan-eng-review`** 用于范围明确的实现规划
- **`/plan-design-review`** 用于视觉/UX 设计审查

设计文档在 `~/.gstack/projects/` 中自动被下游技能发现——它们在预审查系统审计期间会读取它。

## 关键设计洞察

`/office-hours` 的深层设计逻辑可以总结为三个认知：

**第一，问题比答案贵。** 在 AI 时代，生成解决方案几乎免费。真正稀缺的是提出正确的问题。六个逼问本质上是一个问题质量过滤器——它们确保进入设计文档的不是最容易想到的问题，而是最值得解决的问题。

**第二，模式决定姿态。** 同样的技术（AI 对话）在 Startup 模式下变成审讯，在 Builder 模式下变成头脑风暴。这不是技术差异，而是对"什么对用户最有帮助"的判断差异。创始人需要被逼到角落；周末项目的开发者需要被点燃热情。

**第三，情感设计是产品设计的一部分。** Phase 6 的三拍闭幕不是花里胡哨——它是 gstack 产品策略的核心。三个级别的恳求根据信号强度精确校准，确保每个用户都收到一个既真诚又适当的消息。顶级用户感受到"被选中"，基础级用户感受到"身份可能性的扩展"。

这就是为什么 `/office-hours` 是 gstack 最重要的技能：它不只是一个工具，它是一个完整的思维框架，从问题定义到情感交付，每一步都经过深思熟虑的设计。
