# 06. Sprint 流水线与信息流

> 单个 Skill 的价值有限。gstack 的真正力量在于 Skill 之间的信息流：一个 Skill 的输出自动成为下一个 Skill 的输入。设计文档喂给计划评审，测试计划喂给 QA，评审发现喂给发布验证。这不是"工具集"，这是一条流水线。

---

## 一、完整的 Sprint 生命周期

gstack 的 Skill 体系覆盖了从想法到上线的完整工程生命周期。每个阶段对应一个或多个 Skill，它们之间通过持久化文件交换信息：

```
Think          Plan            Build         Review        Test       Ship       Reflect
  │              │               │              │            │          │           │
/office-hours  /plan-ceo-review  [coding]     /review      /qa       /ship      /retro
               /plan-eng-review              /codex       /qa-only
               /plan-design-review           /design-review
```

信息在这条流水线中单向流动，但每个节点都可以独立调用。你可以跳过 CEO Review 直接做 Eng Review，可以不做 QA 直接 Ship——但每个 Skill 会检查上游输出是否存在，并据此调整自己的行为。

### 阶段详解

**Think 阶段**：`/office-hours` 是起点。它通过六个逼问式问题（你的用户真的愿意为此付费吗？你见过的需求证据是什么？最窄的楔子是什么？）把模糊的想法塑造成清晰的设计文档。

**Plan 阶段**：三个评审 Skill 从不同角度审视计划。CEO Review 关注商业价值和战略一致性；Eng Review 关注架构可行性和技术债务；Design Review 关注用户体验和交互一致性。

**Build 阶段**：这是用户自己写代码的阶段。gstack 在此阶段不主动介入，但浏览器引擎（`/browse`）随时可用于验证。

**Review 阶段**：`/review` 做 Claude 自己的代码审查，`/codex` 提供 OpenAI 模型的独立第二意见。两者可以交叉对比。

**Test 阶段**：`/qa` 打开浏览器做端到端测试，`/qa-only` 只报告不修复。

**Ship 阶段**：`/ship` 是全自动流水线——合并基础分支、运行测试、生成版本号、写 CHANGELOG、创建 PR。

**Reflect 阶段**：`/retro` 分析 commit 历史、工作模式、代码质量趋势，生成团队回顾报告。

## 二、跨 Skill 信息交换：项目目录

gstack 的跨 Skill 信息交换不依赖数据库、不依赖 API、不依赖内存——它使用文件系统。

每个项目在 `~/.gstack/projects/{slug}/` 下有自己的目录：

```
~/.gstack/projects/
└── my-app/                          # 项目 slug（从 git remote 推导）
    ├── garry-feat-auth-design-20260315-142300.md   # 设计文档
    ├── garry-feat-auth-design-20260316-091200.md   # 修订版设计文档
    ├── feat-auth-reviews.jsonl       # 评审日志
    ├── ceo-plans/                    # CEO Review 产出
    │   ├── 20260316-auth-strategy.md
    │   └── archive/                  # 已归档的旧计划
    └── test-outcomes/                # 测试结果
```

项目 slug 由 `bin/gstack-slug` 脚本从 git remote URL 自动推导。这确保了无论用户在哪个目录下工作，只要是同一个 Git 仓库，就会共享同一套项目数据。

## 三、设计文档的修订链

`/office-hours` 每次运行都会生成一个设计文档。当在同一分支上多次运行时，它会创建修订链：

```bash
# office-hours/SKILL.md.tmpl 中的设计谱系检测
PRIOR=$(ls -t ~/.gstack/projects/$SLUG/*-$BRANCH-design-*.md 2>/dev/null | head -1)
```

如果存在前一版设计文档，新文档会包含 `Supersedes:` 字段：

```markdown
# Design: Auth Flow Redesign

Generated by /office-hours on 2026-03-16
Branch: feat-auth
Status: DRAFT
Mode: Startup
Supersedes: garry-feat-auth-design-20260315-142300.md

## Problem Statement
...
```

下游 Skill 会利用这个修订链。例如 `/plan-eng-review` 在读到设计文档时：

```
If a design doc exists, read it. Use it as the source of truth for the
problem statement, constraints, and chosen approach. If it has a
`Supersedes:` field, note that this is a revised design — check the prior
version for context on what changed and why.
```

这意味着 Eng Review 不仅知道当前设计是什么，还知道设计是如何演变的——哪些假设被推翻了，哪些约束被放宽了。这种上下文在纯人工流程中往往会丢失。

## 四、评审日志与 Review Readiness Dashboard

gstack 的评审系统有一个精巧的持久化层。每次评审（无论是 Eng Review、CEO Review、Design Review 还是 Codex Review）的结果都会以 JSONL 格式追加到评审日志中：

```bash
#!/usr/bin/env bash
# bin/gstack-review-log — atomically log a review result
set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
eval $("$SCRIPT_DIR/gstack-slug" 2>/dev/null)
GSTACK_HOME="${GSTACK_HOME:-$HOME/.gstack}"
mkdir -p "$GSTACK_HOME/projects/$SLUG"
echo "$1" >> "$GSTACK_HOME/projects/$SLUG/$BRANCH-reviews.jsonl"
```

每条评审记录是一个 JSON 对象：

```json
{"skill":"plan-eng-review","timestamp":"2026-03-16T15:00:00Z","status":"clean","commit":"a1b2c3d"}
{"skill":"codex-review","timestamp":"2026-03-16T16:30:00Z","status":"issues_found","gate":"fail","findings":3}
{"skill":"ship-review-override","timestamp":"2026-03-16T17:00:00Z","decision":"ship_anyway"}
```

JSONL 格式（每行一个 JSON）的选择不是偶然的：它支持原子追加（`>>`），不需要读取整个文件就能添加新记录，天然抗并发冲突。

### Review Readiness Dashboard

`/ship` 在执行前会调用 `gstack-review-read` 读取评审日志，并渲染一个 Review Readiness Dashboard：

```
+====================================================================+
|                    REVIEW READINESS DASHBOARD                       |
+====================================================================+
| Review          | Runs | Last Run            | Status    | Required |
|-----------------|------|---------------------|-----------|----------|
| Eng Review      |  1   | 2026-03-16 15:00    | CLEAR     | YES      |
| CEO Review      |  0   | —                   | —         | no       |
| Design Review   |  0   | —                   | —         | no       |
| Codex Review    |  1   | 2026-03-16 16:30    | ISSUES    | no       |
+--------------------------------------------------------------------+
| VERDICT: CLEARED — Eng Review passed                                |
+====================================================================+
```

Dashboard 的判定逻辑：

- **CLEARED**：Eng Review 在 7 天内有记录且状态为 "clean"（或全局配置 `skip_eng_review` 为 true）
- **NOT CLEARED**：Eng Review 缺失、过期（>7 天）或有未解决问题
- CEO、Design、Codex Review 仅供参考，永远不阻断发布

### 陈旧性检测

Dashboard 不仅检查评审是否存在，还检查评审是否仍然有效。每条评审记录包含当时的 commit hash：

```
对于每条有 commit 字段的评审记录：将其与当前 HEAD 对比。
如果不同，计算间隔的 commit 数量：
  git rev-list --count STORED_COMMIT..HEAD
显示："Note: Eng Review from 2026-03-16 may be stale — 5 commits since review"
```

这解决了一个常见问题：评审通过后又做了改动。在传统流程中，这种"评审漂移"几乎无法追踪。gstack 通过 commit hash 比对让它显式可见。

## 五、评审门控与覆盖机制

`/ship` 的评审门控不是简单的"通过/不通过"。它有一套精心设计的覆盖机制：

```markdown
If the Eng Review is NOT "CLEAR":

1. Check for a prior override on this branch:
   grep '"skill":"ship-review-override"' ~/.gstack/projects/$SLUG/$BRANCH-reviews.jsonl

2. If an override exists: "Review gate previously accepted — continuing."

3. If no override exists, AskUserQuestion:
   A) Ship anyway
   B) Abort — run /plan-eng-review first
   C) Change is too small to need eng review

4. If A or C: persist the decision to review log
```

关键设计：用户的覆盖决策会被持久化到评审日志中。这意味着在同一分支上再次运行 `/ship` 时，不会重复询问。这是"不重复询问同一个问题"原则的体现。

## 六、智能路由：不是每个变更都需要所有评审

gstack 不会机械地要求所有评审。它有"智能路由"逻辑：

**CEO Review**：推荐用于重大产品/业务变更、新的用户可见功能、范围决策。对于 bug 修复、重构、基础设施、清理工作，跳过。

**Design Review**：检测 diff 是否涉及前端代码：

```bash
source <(~/.claude/skills/gstack/bin/gstack-diff-scope <base> 2>/dev/null)
# 如果 SCOPE_FRONTEND=true 且没有设计评审记录...
```

只有当变更涉及前端代码时，才建议运行 Design Review。纯后端变更不需要。

**Codex Review**：只有当 Codex CLI 已安装时才显示。它是一个附加视角，不是必需流程。

这种路由逻辑确保了流水线不会变成官僚流程——每个评审只在有价值时才出现。

## 七、跨会话上下文传递

gstack 的一个核心挑战是：Claude Code 的会话是短暂的。每次启动新会话，AI 不记得上一次的对话。gstack 通过文件系统解决这个问题：

```
会话 1: /office-hours
  └─ 写入设计文档 → ~/.gstack/projects/my-app/garry-feat-auth-design-*.md

会话 2: /plan-ceo-review
  └─ 读取设计文档 → 作为评审输入
  └─ 写入 CEO 计划 → ~/.gstack/projects/my-app/ceo-plans/

会话 3: /plan-eng-review
  └─ 读取设计文档 + CEO 计划 → 作为评审输入
  └─ 写入评审记录 → ~/.gstack/projects/my-app/feat-auth-reviews.jsonl

会话 4: /ship
  └─ 读取评审日志 → 渲染 Dashboard → 决定是否放行
```

每个会话都是无状态的，但通过读写 `~/.gstack/projects/` 目录中的文件，它们形成了一条有状态的流水线。文件系统就是消息总线。

### gstack-slug：项目身份标识

跨会话上下文的前提是准确识别"当前项目"。`gstack-slug` 脚本从 Git remote URL 推导出一个稳定的项目标识符：

```bash
source <(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)
# 输出: SLUG=my-org-my-app  BRANCH=feat-auth
```

无论用户在项目的哪个子目录中，`gstack-slug` 都会通过 `git rev-parse --show-toplevel` 和 remote URL 推导出相同的 slug。这确保了 `/office-hours` 在 `~/code/my-app/src/` 下写的设计文档，`/ship` 在 `~/code/my-app/` 下也能找到。

## 八、信息流的完整图景

让我们用一个具体场景来展示完整的信息流。假设你要为一个 SaaS 应用添加支付功能：

```
/office-hours "I want to add Stripe payments"
  │
  │  输出: ~/.gstack/projects/saas-app/garry-main-design-20260316-100000.md
  │        （包含问题陈述、需求证据、三种实现方案、推荐方案）
  │
  ▼
/plan-ceo-review
  │  读取: 设计文档
  │  输出: ~/.gstack/projects/saas-app/ceo-plans/20260316-stripe-payments.md
  │  输出: ...reviews.jsonl += {"skill":"plan-ceo-review","status":"clean",...}
  │
  ▼
/plan-eng-review
  │  读取: 设计文档 + CEO 计划
  │  输出: ...reviews.jsonl += {"skill":"plan-eng-review","status":"clean",...}
  │
  ▼
[用户编写代码，创建 feat-stripe 分支]
  │
  ▼
/review
  │  读取: TODOS.md + git diff
  │  检查: 范围漂移（是否实现了计划之外的东西？）
  │  输出: ...reviews.jsonl += {"skill":"review","status":"issues_found",...}
  │
  ▼
/codex review
  │  读取: git diff（独立于 /review）
  │  输出: GATE: PASS / FAIL
  │  输出: 交叉模型分析（Claude 和 Codex 各发现了什么）
  │  输出: ...reviews.jsonl += {"skill":"codex-review","gate":"pass",...}
  │
  ▼
/qa
  │  读取: git diff（识别受影响的页面/路由）
  │  执行: 浏览器端到端测试
  │  输出: QA 报告 + 截图证据
  │
  ▼
/ship
  │  读取: 评审日志 → 渲染 Dashboard
  │  验证: Eng Review CLEARED
  │  执行: 合并基础分支 → 运行测试 → 版本号 → CHANGELOG → PR
  │
  ▼
/document-release
  │  读取: git diff + PR body
  │  更新: README、API 文档、用户指南
  │
  ▼
/retro
     读取: commit 历史 + 工作模式
     输出: 回顾报告 + 趋势分析
```

这条流水线中，没有任何手动的"把这个文件传给那个工具"的操作。每个 Skill 知道去哪里找上游的输出，知道把自己的输出写到哪里。用户只需要按顺序调用 Skill（甚至不需要严格按顺序——跳过步骤也是合法的），信息会自动流动。

## 九、为什么不用数据库

一个常见的架构直觉是：跨组件状态管理应该用数据库。为什么 gstack 选择了文件系统？

1. **零依赖**：不需要安装、配置或运行数据库服务。`~/.gstack/` 目录就是"数据库"。
2. **人类可读**：用户可以直接 `cat` 任何文件查看内容。调试时不需要查询工具。
3. **与 Git 协作**：文件天然支持 diff、备份、迁移。JSONL 格式支持原子追加。
4. **跨工具兼容**：任何语言都能读写文件。bash 脚本、TypeScript、Python 都行。
5. **离线优先**：不需要网络连接。所有数据在本地。

这种选择反映了 gstack 的一个核心设计原则：**用最简单的技术解决问题**。文件系统是最古老、最可靠、最普遍的"数据库"。在 AI Agent 的场景下——每次读写都是完整的文件操作，没有复杂的查询需求——它恰好是最优解。
