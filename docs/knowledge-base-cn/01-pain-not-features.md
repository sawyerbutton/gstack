# 01. 从痛苦出发，不从功能出发

> 用户说"日程简报 app"。AI 说"你在描述的是一个私人幕僚长"。——这不是 AI 在炫技，是一种经过编码的思维方式在起作用。

---

## 为什么这篇文章最重要

如果你只读知识库中的一篇文章，应该是这篇。不是因为它最复杂，而是因为它包含了 gstack 最根本的认知转变：

> **你描述的"功能"几乎从来不是你真正需要的东西。你的痛苦才是。**

这个洞察不是 gstack 发明的——它是 YC 二十年来反复验证的产品真理。gstack 做的事情是把它**编码成了可执行的 AI 行为**。

---

## 一个真实案例的完整拆解

### 用户说了什么

> "我想做一个日程简报 app，管理多个 Google 日历。"

这听起来很明确。大多数 AI 助手会立刻开始：
- 设计日历 API 集成
- 写一个简报模板
- 搭建前端页面

### `/office-hours` 做了什么

它没有开始写代码。它问了一个问题：

> "说说你的痛苦——具体的例子，不是假设。"

用户描述了：
- 助理漏掉了跨多个 Google 账号的事件
- 日历里有过时的信息、错误的地点
- 准备文档是 AI slop（低质量 AI 生成内容）
- 追踪错误地点花费大量时间

然后 `/office-hours` 做了一件关键的事——**重新定义问题**：

> "我要推回你的框架。你说的是'日程简报 app'。但你实际描述的是一个**私人幕僚长 AI**。"

它提炼出用户自己没有意识到的五个能力需求：

1. **监控日历**（跨账号检测过时信息、缺失地点、权限问题）
2. **生成真正的准备工作**（不是物流摘要，是为董事会会议做的智力准备）
3. **管理 CRM**（你在见谁、关系是什么、他们想要什么）
4. **优先化时间**（标记需要提前准备的事件、主动阻塞时间）
5. **用钱换杠杆**（主动寻找可以委托或自动化的方式）

### 为什么这个重新定义很重要

用户本来要做的是一个**每天发邮件的日程简报**。现在他要做的是一个**价值十倍的私人幕僚长**——而且最窄楔子（narrowest wedge）仍然是从日程简报开始，只是方向完全不同了。

**关键区别：** 如果直接按"日程简报"来做，三个月后你有一个日程简报 app。如果按"私人幕僚长"来做，三个月后你有一个日程简报 app + CRM 集成 + 智能准备 + 时间优先化的雏形——因为从第一天起架构就指向了正确的方向。

## 六个强制性问题的底层逻辑

`/office-hours` 的创业模式包含六个问题。每个问题都经过精心设计，不是为了收集信息，而是为了**暴露创始人思维中的盲点**。

### Q1：需求真实性

**问的是：** "你有什么最强的证据证明有人真的想要这个——不是'感兴趣'，不是'注册了候补名单'，而是如果明天它消失了会真正感到不安？"

**设计意图：** 区分"兴趣"和"需求"。

这个区分是 YC 二十年来最核心的一课：

| 表现 | 看起来像 | 实际是 |
|------|---------|--------|
| 500 人注册候补名单 | 需求 | 好奇心 |
| VC 对这个领域很兴奋 | 市场验证 | 别人的赌注 |
| 用户说"这很有趣" | 产品-市场匹配 | 礼貌 |
| 用户在你宕机 20 分钟时打电话来 | 依赖 | **真正的需求** |
| 用户围绕你的产品重建了工作流 | 嵌入 | **真正的需求** |

**危险信号检测：** 如果创始人的回答里出现"人们说很有趣"、"我们有 500 个候补名单注册"、"VC 对这个领域很兴奋"——`/office-hours` 会直接指出这些都不是需求。

### Q2：现状

**问的是：** "你的用户现在怎么解决这个问题——即使很差？这个变通方案让他们付出了什么代价？"

**设计意图：** 找到真正的竞争对手。

> 你真正的竞争对手不是另一家创业公司。是用户现在凑合用的 spreadsheet + Slack + 手动操作。

**核心洞察：** 如果"什么都没有"是当前的解决方案——如果真的没有人在做任何事情来解决这个问题——这通常意味着**问题还不够痛**。足够痛的问题总是有变通方案，哪怕是很糟糕的变通方案。

**追问目标：** 一个具体的工作流。花费的小时数。浪费的金钱。拼凑在一起的工具。为了手动完成这件事而雇佣的人。

### Q3：极致具体

**问的是：** "说出最需要这个东西的那个人。他们的职位是什么？什么让他们升职？什么让他们被解雇？什么让他们夜不能寐？"

**设计意图：** 从"类别"降级到"个人"。

这个问题的心理学原理是**具体性强制清晰思考**。当你说"企业客户"时，你的大脑在做模糊推理。当你说"Sarah，Acme Corp 的运营经理，每周花 15 小时手动整理发货数据，如果出错会被她的 VP 骂"时，你的大脑被迫做精确推理。

**危险信号：** 类别级别的回答。"医疗保健企业。""中小企业。""营销团队。"这些是筛选器，不是人。你不能给一个类别发邮件。

### Q4：最窄楔子

**问的是：** "什么是这个东西的最小版本，让人会为此本周就掏真金白银——不是在你建完平台之后？"

**设计意图：** 打破"先建平台再做验证"的思维定势。

**核心原则：** 早期窄 > 早期宽。

```
传统思维:  构思完整愿景 → 建造完整平台 → 找到用户 → 发现没人需要
YC 思维:   找到痛苦 → 最窄楔子 → 本周有人付钱 → 从强项扩展
```

**追问目标：** 一个功能。一个工作流。也许只是一封每周邮件或一个自动化。可以在几天内发布、有人会付钱的东西。

**危险信号：** "我们需要先建完整平台才能让人使用。""我们可以精简但那样就没有差异化了。"——这些是对架构的迷恋，而不是对价值的追求。

**杀手追问：** "如果用户完全不需要做任何事情就能获得价值呢？不需要登录、不需要集成、不需要设置。那会是什么样子？"

### Q5：观察与惊讶

**问的是：** "你有没有坐下来看过用户使用这个产品——不帮忙？他们做了什么让你惊讶的事？"

**设计意图：** 检验创始人是否在"看"还是在"想象"。

> 引导式演示教会你的东西为零。坐在某人身后看他们挣扎——同时咬住自己的舌头——教会你一切。

**黄金信号：** 用户在做产品设计之外的事情。这通常是真正的产品试图浮现。

**危险信号：** "我们做了调查问卷。""我们做了一些演示。""没有什么意外，一切按预期进行。"——调查问卷会撒谎，演示是剧场，"按预期"意味着通过现有假设过滤。

### Q6：未来适配

**问的是：** "如果三年后世界看起来有意义的不同——它会的——你的产品会变得更重要还是更不重要？"

**设计意图：** 区分"搭顺风车"和"有独特论点"。

**危险信号：** "市场年增长 20%"——增长率不是愿景。"AI 会让一切变好"——这是涨潮论点，每个竞争对手都能说。

**追问目标：** 一个具体的主张——关于用户世界如何变化，以及为什么这种变化让他们的产品更有价值。

---

## 智能路由：不是每次都问全部六个

这是设计上的精妙之处。不同阶段的产品需要不同的问题：

| 产品阶段 | 问哪些 | 为什么 |
|---------|--------|--------|
| 想法阶段（无用户）| Q1, Q2, Q3 | 验证需求是否真实 |
| 有用户（未付费）| Q2, Q4, Q5 | 找到付费楔子 |
| 有付费用户 | Q4, Q5, Q6 | 扩展和未来化 |
| 纯工程/基础设施 | Q2, Q4 | 实用性导向 |

如果用户前面的回答已经覆盖了后面的问题，跳过。不问已经清楚的东西。

## "逃生舱口"设计

gstack 不是死板的流程。如果用户说"直接做"、表现出不耐烦、或者提供了一个已经成型的计划——快进到替代方案生成阶段。

但即使跳过了问题阶段，**前提挑战（Phase 3）和替代方案生成（Phase 4）仍然是强制的**。

为什么？因为即使"简单"的计划也能从前提检验和强制替代方案中受益。你可能已经想清楚了方向，但你可能没有考虑过第二种实现方式。

---

## 运作原则的深层解读

### "具体性是唯一货币"

这不只是"请说具体点"。这是一种认知纪律。

当有人说"我们的目标客户是企业"时，他们的大脑没有在做真正的推理——他们在做**标签匹配**。"企业"不是一个可以服务的实体，它是一个分类学上的容器。

当你逼迫他们说出"Sarah，50 人物流公司的运营经理"时，你逼迫他们从标签匹配切换到**场景推演**。这种认知模式的切换才是具体性的真正价值。

### "现状是你真正的竞争对手"

大多数创始人在谈竞争时想的是"我们 vs 另一家创业公司"。但 YC 二十年的数据显示，真正的竞争对手是**惯性**——用户当前解决问题的方式，哪怕这种方式很糟糕。

如果用户现在用 Excel + 邮件 + 手动操作来完成一个流程，而你的产品只比这好一点点，他们不会切换。切换成本太高了。你的产品必须比他们的变通方案好 **10 倍**才能让他们动起来。

这就是为什么 Q2 如此关键——它不仅帮你理解竞争，更帮你校准"好多少才够"。

### "观察，不要演示"

这个原则背后是一个认知偏见：**确认偏误**（confirmation bias）。

当你演示产品时，你控制了叙事。用户顺着你的引导走。他们点你让他们点的按钮，看你让他们看的功能。你学到的是"我的演示脚本有没有逻辑漏洞"，不是"用户真的怎么用这个东西"。

当你坐在用户身后、闭上嘴时，你看到的是：
- 他们先看向哪里（不是你认为重要的地方）
- 他们在哪里卡住了（不是你觉得复杂的地方）
- 他们试图做什么你没设计的事（这可能才是真正的产品）

## 前提挑战：在提出解决方案之前

六个问题之后，`/office-hours` 不会直接跳到"那我们怎么做"。它会先挑战前提：

1. **这是正确的问题吗？** 不同的框架能否带来戏剧性更简单或更有影响力的解决方案？
2. **如果什么都不做会怎样？** 是真正的痛点还是假设性的？
3. **现有代码已经部分解决了什么？** 映射可复用的模式。

前提以可否证的陈述形式呈现：

```
前提：
1. 日历是锚定数据源，但价值在上面的智能层 — 同意/不同意？
2. 助理不会被替代，而是被增强 — 同意/不同意？
3. 最窄楔子是一个真正有效的日程简报 — 同意/不同意？
```

**每个你接受的前提都成为设计文档中的承重结构。** 如果以后发现某个前提是错的，你知道要重新检查哪些决策。

## 强制性替代方案

这是 `/office-hours` 和所有 plan review skill 共享的核心机制：

> **每个设计都必须考虑替代方案。这不是可选的。**

至少 2 个方案，最好 3 个。规则：
- 一个必须是**最小可行版**（最少文件、最小 diff、最快发布）
- 一个必须是**理想架构**（最佳长期轨迹、最优雅）
- 一个可以是**创造性/横向的**（意想不到的方式、不同的问题框架）

为什么？因为第一个想到的方案几乎总是"最显而易见的"。强制生成替代方案是打破思维惯性的机制。

## 从 `/office-hours` 中可以带走什么

即使你不使用 gstack，以下习惯可以立刻改善你的产品决策：

1. **在写代码之前问"痛苦是什么"** — 不接受功能请求本身作为规格
2. **追问到具体的人** — "谁需要这个？叫什么名字？什么让他们夜不能寐？"
3. **找到现状对手** — "他们现在怎么做？这个变通方案花了他们什么？"
4. **强制替代方案** — 每次重要决策都问"还有什么其他方式可以做到这一点？"
5. **挑战前提** — 在提出解决方案之前，把你的假设写成可否证的陈述
6. **最窄楔子** — "本周有人会为此付钱的最小版本是什么？"

这六个习惯来自 YC 二十年的创业辅导经验。gstack 只是把它们变成了可以自动运行的 prompt。
