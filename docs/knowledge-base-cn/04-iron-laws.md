# 04. 铁律系统：用约束驱动质量

> "应该能工作了"不是证据。运行测试，贴出结果。——`/ship` 验证门

---

## 为什么需要铁律

AI agent 有两个系统性弱点：

1. **过度自信**：AI 很容易说"这应该修好了"而不实际验证。它在生成代码时的自信和代码实际正确性之间存在危险的差距。

2. **范围蔓延**：给 AI 一个 bug 修复任务，它可能会顺便"改进"周围的代码、添加"更好的"错误处理、重构一个"不够优雅"的函数——每一个额外的改动都引入新的风险。

gstack 的解决方案不是更好的 prompt engineering——而是**铁律**：一组不可违背的约束，嵌入在 skill 的工作流中，无论 AI 多么"自信"都必须遵守。

---

## 五条核心铁律

### 铁律一：没有根因就没有修复

> **出处：** `/investigate` — "NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST."

**完整表述：** 修复症状会创造打地鼠式的调试。每个不针对根因的修复都让下一个 bug 更难找到。找到根因，然后修复它。

**实践机制：**

```
Phase 1: 根因调查
  ├── 收集症状（错误消息、堆栈追踪、复现步骤）
  ├── 读代码（从症状追溯到可能的原因）
  ├── 检查最近变更（git log -- <affected-files>）
  └── 复现（能否确定性地触发？）

Phase 2: 模式分析
  ├── 竞态条件？→ 检查并发共享状态
  ├── Nil 传播？→ 检查可选值的守卫
  ├── 状态损坏？→ 检查事务、回调
  ├── 集成失败？→ 检查外部 API 调用
  └── 配置漂移？→ 检查环境变量、特性标志

Phase 3: 假设测试
  ├── 添加临时日志/断言验证假设
  ├── 如果假设错误 → 回到 Phase 1
  └── 3 次假设失败 → 强制停止，升级

Phase 4: 实现（仅在根因确认后）
  ├── 最小修复
  ├── 回归测试
  └── 运行全套测试
```

**3 次法则：** 如果 3 个假设都失败了，`/investigate` 强制停止并问用户：

```
3 个假设都已测试，没有匹配。这可能是架构问题而非简单 bug。

A) 继续调查——我有新假设：[描述]
B) 升级给人类审查——需要了解系统的人
C) 添加日志等待——插桩观测区域，下次捕获
```

**红旗检测：**
- "先临时修一下" → 没有"临时"。修好或升级。
- 在追踪数据流之前就提出修复方案 → 你在猜。
- 每次修复都暴露新问题 → 错误的层，不是错误的代码。

### 铁律二：没有验证就没有完成声明

> **出处：** `/ship` Step 6.5 — "IRON LAW: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE."

**完整表述：**

```
声称完成但没有验证 = 不诚实，不是效率。

防合理化规则：
  "应该能工作了"    → 运行它。
  "我很自信"        → 自信不是证据。
  "我之前已经测试过" → 代码在那之后变了。再测。
  "这是微不足道的变更" → 微不足道的变更搞垮生产。
```

**实践机制：** `/ship` 在 push 之前检查：如果在 Step 3 的测试之后有**任何代码变更**（review 修复、CHANGELOG 不算），必须重新运行测试套件。不接受旧的测试输出。

**为什么这条铁律如此重要：** AI 的一个核心弱点是"合理化"——它可以为几乎任何结论构建看似合理的论证。"这个改动不需要重新测试因为它只改了注释"听起来很合理，但如果注释改动触发了 linter 规则变更呢？唯一不会被合理化攻破的防线是**绝对规则**：变了 → 测。没有例外。

### 铁律三：每次修复一个 commit

> **出处：** `/qa` — "One commit per fix. Never bundle multiple fixes."

**完整表述：** QA 阶段的每个 bug 修复都是独立的 atomic commit。格式：`fix(qa): ISSUE-NNN — short description`。

**为什么：**
- 如果修复 B 引入了回归，可以独立 revert 而不影响修复 A
- `git bisect` 可以精确定位每个问题
- 代码审查可以逐个修复评估

**范围控制：** 每个修复只修改**直接相关的文件**。不重构周围代码，不添加功能，不"改进"无关的东西。

### 铁律四：沉默失败是关键缺陷

> **出处：** `/plan-ceo-review` — "Zero silent failures. Every failure mode must be visible."

**完整表述：**

```
失败模式注册表：
代码路径 | 失败模式    | 被救？ | 有测试？ | 用户看到？ | 有日志？
---------|------------|--------|---------|-----------|--------
...      | ...        | N      | N       | 静默      | N

任何一行 被救=N 且 测试=N 且 用户看到=静默 → **关键缺陷**
```

**实践含义：** `/plan-ceo-review` 的 Section 2 要求为**每个可能失败的方法**填写完整的错误映射表。不是"处理错误"这样的模糊声明——而是：

- 这个方法有什么可以出错？（具体异常类）
- 被捕获了吗？（是/否）
- 捕获后做了什么？（重试/优雅降级/重新抛出？）
- 用户看到了什么？
- Catch-all 错误处理（`rescue StandardError`、`catch Exception`）总是气味——必须标记

### 铁律五：不要在没有收据的情况下说"已存在的问题"

> **出处：** CLAUDE.md — E2E eval failure blame protocol

**完整表述：**

> 当 E2E 评估在 `/ship` 或任何其他工作流中失败时，**永远不要在没有证明的情况下声称"与我们的变更无关"**。这些系统有隐形耦合——preamble 文本变更影响 agent 行为，新 helper 改变时序，重新生成的 SKILL.md 偏移 prompt 上下文。
>
> **在归因为"已存在"之前必须：**
> 1. 在 main 分支上运行相同的 eval，证明它在那里也失败
> 2. 如果在 main 通过但在分支上失败——这就是你的变更。追踪归因。
> 3. 如果无法在 main 上运行，说"未验证——可能与变更相关也可能不相关"，并在 PR 中标记为风险
>
> "已存在"没有收据就是一种懒惰的声称。证明它或不要说它。

## 约束如何驱动质量

### 原理：减少决策点

每个铁律的作用是**消除一个决策点**。

没有铁律时，AI 在每个修复后面临一个决策："要不要验证？"这个决策取决于修复的大小、自信程度、时间压力等因素。而这种情况下，AI 的"判断"往往倾向于跳过验证——因为验证需要时间，而 AI 的生成模式天然偏好"继续前进"。

有铁律时，决策被消除：**变了 → 测。** 没有"判断"的空间。

### 原理：对称的约束

注意铁律是对称的——它们不仅约束 AI，也约束人类：

| 铁律 | 约束 AI 的场景 | 约束人类的场景 |
|------|---------------|---------------|
| 没有根因就没有修复 | 防止 AI 在症状上打补丁 | 防止人类说"先试试这个" |
| 没有验证就没有完成 | 防止 AI 说"应该能工作" | 防止人类说"看起来对" |
| 每次修复一个 commit | 防止 AI 捆绑"顺便改进" | 防止人类说"一起提交吧" |
| 沉默失败是缺陷 | 防止 AI 写 catch-all 吞错误 | 防止人类说"这个错误很罕见" |

### 原理：WTF 停止机制

`/qa` 的 WTF-likelihood 计算是一种独特的"自动熔断器"：

```
WTF-LIKELIHOOD:
  起始 0%
  每次回滚:                +15%
  每个修复触及 >3 文件:     +5%
  修复 15 次之后:           每次额外 +1%
  所有剩余低优先级:         +10%
  触及无关文件:             +20%
```

**WTF > 20% → 立即停止。** 硬上限：50 次修复。

**为什么这个机制很聪明：**

这不是一个简单的计数器。它编码了一种直觉——"事情正在变得不对劲"的感觉。

- 回滚意味着修复引入了回归 → 你可能不理解系统
- 触及 >3 文件意味着修复影响范围大 → 可能是架构问题
- 修复数量过多意味着你可能在打地鼠
- 触及无关文件意味着范围蔓延

每个信号单独来看不算严重，但它们的累积效应表明：**你正在做的事情越来越不可控**。WTF 机制在局面失控之前强制叫停。

## 铁律在不同 Skill 中的体现

### `/investigate` 的范围锁定

调试时，`/investigate` 在形成根因假设后会自动锁定编辑范围：

```bash
echo "src/auth/" > ~/.gstack/freeze-dir.txt
echo "Debug scope locked to: src/auth/"
```

所有后续的 Edit 和 Write 操作如果目标文件不在 `src/auth/` 下，会被 hook **拦截**（不是警告，是拦截）。

**为什么：** 调试 auth 模块时，AI 可能"发现" payment 模块也有类似问题然后去修——这会让调试范围失控。范围锁定消除了这种可能。

### `/ship` 的 Review Readiness Dashboard

在 push 之前，`/ship` 检查所有已运行的 review 的状态：

- 工程 review 是否通过？（必须）
- CEO review 是否过时？（commit hash 漂移检测）
- 设计 review 是否存在？（如果有前端变更）
- Codex review 的门控结果？（如果运行了）

如果工程 review 不是 CLEAR 状态，`/ship` 不会静默继续——它会问用户做出明确决策，并持久化这个决策以便后续 `/ship` 运行不会重复询问。

### `/qa` 的 clean working tree 要求

`/qa` 在开始之前强制要求 clean working tree：

```
你的工作树有未提交的变更。/qa 需要 clean tree 以便每个 bug 修复
获得自己的 atomic commit。

A) 提交我的变更 — 用描述性消息提交所有当前变更，然后开始 QA
B) Stash 我的变更 — stash，运行 QA，之后 pop
C) 中止 — 我手动清理
```

这不是技术限制——git 完全可以在 dirty tree 上工作。这是一种**纪律约束**：确保 QA 阶段的每个 commit 都是一个独立的、可追踪的 bug 修复。

## 设计你自己的铁律

如果你想在自己的团队或工作流中引入铁律，以下是 gstack 的设计模式：

### 一条好的铁律需要什么

1. **简单到不需要判断**：如果规则需要"根据情况判断"，它就不是铁律——它只是一个建议。
2. **有明确的触发条件**：什么时候这条规则生效？
3. **有明确的行动**：违反时做什么？
4. **对称适用**：对 AI 和人类同样有效。
5. **有逃生舱口**：在真正例外的情况下，用户可以 override——但必须明确选择。

### 铁律 vs 指南

| 特征 | 铁律 | 指南 |
|------|------|------|
| 违反后果 | 工作流停止 | 警告，继续 |
| 判断空间 | 零 | 需要权衡 |
| 例外处理 | 明确的用户 override | 自行判断 |
| 示例 | "变了→测" | "保持函数在 50 行以内" |

### 推荐的团队铁律（灵感来自 gstack）

1. **PR 不能没有测试**：如果 diff 包含业务逻辑变更，必须有对应的测试变更。没有例外。
2. **部署后 5 分钟验证**：每次部署后必须运行验证检查清单。"看起来没问题"不算。
3. **根因文档**：每个生产事故修复必须附带根因文档。"修了一个 bug"不是根因。
4. **一个 PR 一个目的**：PR 不能同时包含功能 + 重构 + bug 修复。拆开。

## 铁律的心理学

铁律最深层的价值不是技术性的——而是心理性的。

### 消除"这次特殊"心理

没有铁律时，每个决策都是一次判断调用。"这次测试不够重要可以跳过"、"这个修复太明显不需要验证"、"这个范围扩展只是微小的改进"。每一个单独来看都是"合理"的。累积起来就是质量崩塌。

铁律消除了"这次特殊"的可能。不是因为这次不特殊——而是因为**判断这次是否特殊的成本高于一律执行的成本**。

### 建立信任

当团队知道"变了→测"是绝对规则时，他们对代码库的信任显著提高。不是因为每个测试都完美——而是因为**有一个已知的最低标准**。

这就是铁律的终极价值：它们把"我们的代码可能没问题"变成"我们的代码通过了这些检查"——前者是猜测，后者是事实。
