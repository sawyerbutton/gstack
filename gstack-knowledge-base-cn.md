## 1. 核心理念

### 1.1 不是工具集，是流程

gstack 最容易被误解的地方是：它**不是一堆独立的 AI 工具**。它是一套**完整的软件开发流程**，用 Markdown 文件编码了一个虚拟工程团队的协作方式。

传统的 AI 编程助手是"你问我答"。gstack 的做法是把 AI 变成**15 个不同角色的专家**：

| 角色 | 对应 Skill | 思维模式 |
|------|-----------|---------|
| YC 合伙人 | `/office-hours` | 在写代码之前，先理解你到底在解决什么问题 |
| CEO / 创始人 | `/plan-ceo-review` | 这个需求里藏着什么 10 倍产品？ |
| 工程经理 | `/plan-eng-review` | 让设想变得可构建 |
| 高级设计师 | `/plan-design-review` | 每个设计维度 0-10 评分 |
| 设计合伙人 | `/design-consultation` | 从零构建完整设计系统 |
| Staff Engineer | `/review` | 找到能过 CI 但生产环境会爆的 bug |
| 调试专家 | `/investigate` | 没有根因就没有修复 |
| 设计+代码 | `/design-review` | 审计 + 自动修复 |
| QA 主管 | `/qa` | 打开真实浏览器，点击每一个按钮 |
| 发布工程师 | `/ship` | 一个命令完成整个发布 |
| 技术写作 | `/document-release` | 自动同步所有文档 |
| 回顾主持人 | `/retro` | 团队周报、趋势追踪 |

**关键洞察：** 每个角色不仅有专属的工作流程，还有**专属的思维模式和认知模式**。CEO review 会运用 Bezos 的"可逆性决策框架"和 Jobs 的"减法聚焦"；工程 review 会运用 Brooks 的"本质复杂性 vs 偶发复杂性"。这不是花架子——它直接影响 AI 产出的质量。

### 1.2 为什么叫"软件工厂"

Garry Tan 的核心论点：

> "同一个人，不同时代。区别在于工具。"

他在 2013 年构建 Bookface 时全年 772 个 GitHub 贡献。2026 年前三个月已经超过 1237 个。不是因为他变强了，是因为**一个人 + AI + 结构化流程 = 过去 20 人团队的产出**。

gstack 就是这个等式中的"结构化流程"部分。

## 2. 第一原理：AI 时代的工程方法论

### 2.1 完备性原则（Completeness Principle）

这是 gstack 最反直觉但最重要的原则：

> **AI 编码把实现时间压缩了 10-100 倍。完整实现几乎不比偷懒贵。所以永远选完整版。**

传统软件工程的"技术债务"思维建立在一个假设上：人力工时是瓶颈。所以我们经常说"先这样做，以后再优化"。但当 AI 可以在 15 分钟内写完 2000 行测试时，"以后再补测试"就不再是理性选择了。

gstack 的 effort 压缩表：

| 任务类型 | 人类团队 | CC+gstack | 压缩比 |
|---------|---------|-----------|-------|
| 脚手架/模板代码 | 2 天 | 15 分钟 | ~100x |
| 测试编写 | 1 天 | 15 分钟 | ~50x |
| 功能实现 | 1 周 | 30 分钟 | ~30x |
| Bug 修复 + 回归测试 | 4 小时 | 15 分钟 | ~20x |
| 架构/设计 | 2 天 | 4 小时 | ~5x |

**实践含义：** 当一个 plan review 发现"方案 A（完整版，~150 行）vs 方案 B（90%完成度，~80 行）"时，gstack 总是推荐 A。多出的 70 行在 AI 辅助下只需几秒钟。

### 2.2 治理先于速度

gstack 的第二个核心洞察：

> "没有流程的十个 agent 是十个混乱源。有流程——Think, Plan, Build, Review, Test, Ship——每个 agent 都确切知道该做什么、何时停下。"

这解释了为什么 gstack 能同时跑 10-15 个并行 sprint。不是因为技术上多么复杂，而是因为**每个 sprint 都有明确的阶段和交接点**。

### 2.3 铁律（Iron Laws）

gstack 在多个 skill 中嵌入了不可违背的铁律：

1. **没有根因就没有修复**（`/investigate`）：修复症状会让下一个 bug 更难找到。
2. **没有验证就没有完成声明**（`/ship`）："应该能工作了"不是证据。运行测试，贴出结果。
3. **每次修复一个 commit**（`/qa`）：从不把多个修复捆绑在一起。
4. **沉默失败是关键缺陷**（`/plan-ceo-review`）：每个失败模式都必须可见——对系统、对团队、对用户。
5. **不要在没有收据的情况下说"已存在的问题"**（CLAUDE.md）：先在 main 分支上复现，证明它不是你的变更引起的。

### 2.4 平台无关设计

gstack 的 skill 从不硬编码特定框架的命令、文件模式或目录结构。取而代之的是：

1. **读 CLAUDE.md** 获取项目特定配置
2. **如果找不到，问用户**
3. **把答案持久化到 CLAUDE.md**，这样以后再也不用问

这意味着 gstack 可以用于 Rails、Next.js、Flutter、Go 或任何其他技术栈。

## 3. 系统架构：Skill 即 Markdown 的哲学

### 3.1 为什么是 Markdown

gstack 的核心发现极其简洁：**AI agent 的行为由它读到的 prompt 决定**。SKILL.md 文件就是 prompt。不需要编写插件、SDK 或复杂的集成代码——只需要写得好的 Markdown。

```
SKILL.md.tmpl          (人类编写的 prose + 占位符)
       ↓
gen-skill-docs.ts      (从源代码元数据读取)
       ↓
SKILL.md               (已提交，自动生成的部分)
```

### 3.2 模板系统

每个 SKILL.md 实际上由模板（`.tmpl`）生成。模板包含需要人类判断力的工作流、技巧和示例，而占位符在构建时从源代码中填充：

| 占位符 | 来源 | 生成内容 |
|--------|------|---------|
| `{{PREAMBLE}}` | gen-skill-docs.ts | 启动块：更新检查、会话追踪、贡献者模式 |
| `{{BROWSE_SETUP}}` | gen-skill-docs.ts | 浏览器二进制文件发现 + 设置 |
| `{{BASE_BRANCH_DETECT}}` | gen-skill-docs.ts | 动态检测基础分支 |
| `{{QA_METHODOLOGY}}` | gen-skill-docs.ts | /qa 和 /qa-only 共享的 QA 方法论 |
| `{{REVIEW_DASHBOARD}}` | gen-skill-docs.ts | Review Readiness Dashboard |
| `{{TEST_BOOTSTRAP}}` | gen-skill-docs.ts | 测试框架检测与引导 |
| `{{COMMAND_REFERENCE}}` | commands.ts | 浏览器命令参考 |

**为什么提交到 git，而不是运行时生成？** 三个原因：
1. Claude 在 skill 加载时读取 SKILL.md，没有构建步骤
2. CI 可以验证新鲜度（`gen:skill-docs --dry-run` + `git diff --exit-code`）
3. Git blame 可以工作——能看到命令何时添加、在哪个 commit 中

### 3.3 Preamble：每个 Skill 的共同起点

每个 skill 以一个 `{{PREAMBLE}}` 块开始，处理四件事：

1. **更新检查** — 报告是否有新版本
2. **会话追踪** — 当 3+ 会话同时运行时，进入"ELI16 模式"——每个问题都重新交代上下文
3. **贡献者模式** — 自动记录 gstack 自身表现不佳的情况
4. **AskUserQuestion 格式** — 统一格式：上下文、问题、`RECOMMENDATION: Choose X because ___`、字母选项

### 3.4 浏览器架构

这是 gstack 中"硬核技术"的部分。核心洞察：**AI agent 与浏览器交互需要亚秒级延迟和持久状态。**

```
Claude Code                     gstack
─────────                      ──────
                               ┌──────────────────────┐
  Tool call: $B snapshot -i    │  CLI (编译后二进制)    │
  ─────────────────────────→   │  • 读取状态文件        │
                               │  • POST /command      │
                               │    到 localhost:PORT   │
                               └──────────┬───────────┘
                                          │ HTTP
                               ┌──────────▼───────────┐
                               │  Server (Bun.serve)   │
                               │  • 分发命令           │
                               │  • 与 Chromium 通信   │
                               │  • 返回纯文本         │
                               └──────────┬───────────┘
                                          │ CDP
                               ┌──────────▼───────────┐
                               │  Chromium (headless)   │
                               │  • 持久化标签页        │
                               │  • Cookie 保持        │
                               │  • 30分钟空闲超时      │
                               └───────────────────────┘
```

**为什么选 Bun 而不是 Node：**
- `bun build --compile` 生成 ~58MB 的单文件可执行文件，无需 `node_modules`
- 内置 SQLite 支持（Cookie 解密直接读 Chromium 的 SQLite 数据库）
- 原生 TypeScript，无需编译步骤

**Ref 系统**：Agent 通过 `@e1`, `@e2` 这样的引用来操作页面元素，而不是写 CSS selector 或 XPath。这些引用基于 Playwright 的无障碍树（accessibility tree），不修改 DOM，不受 CSP、React hydration 或 Shadow DOM 影响。

## 4. Sprint 流水线

### 4.1 完整流程

```
Think → Plan → Build → Review → Test → Ship → Reflect
  │       │       │       │       │       │       │
  ▼       ▼       ▼       ▼       ▼       ▼       ▼
office  ceo/eng  (你写   review   qa    ship    retro
-hours  review   代码)
```

**关键特性：每个技能知道前一步做了什么。**

- `/office-hours` 写的设计文档 → `/plan-ceo-review` 自动读取
- `/plan-eng-review` 写的测试计划 → `/qa` 自动消费
- `/review` 发现的 bug → `/ship` 验证是否修复
- `/ship` 的版本号 → `/document-release` 自动同步到文档

### 4.2 信息流

gstack 使用 `~/.gstack/projects/{slug}/` 目录作为跨 skill 的信息交换中心：

```
~/.gstack/projects/{repo-slug}/
├── {user}-{branch}-design-{datetime}.md    ← /office-hours 输出
├── {user}-{branch}-test-plan-{datetime}.md ← /plan-eng-review 输出
├── {user}-{branch}-test-outcome-{datetime}.md ← /qa 输出
├── ceo-plans/                              ← /plan-ceo-review 输出
│   ├── {date}-{feature}.md
│   └── archive/
└── {branch}-reviews.jsonl                  ← 所有 review 记录
```

这个设计使得：
- 不同对话之间可以继承上下文
- 多人协作时能发现彼此的设计文档
- 设计文档形成修订链（`Supersedes:` 字段）

## 5. Skill 逐一拆解

### 5.1 `/office-hours` — YC 办公时间（核心灵魂）

**这是那位 UP 主特别推荐的 skill。** 它的威力不在于技术实现，而在于思维方式。

#### 两种模式

**创业模式（Startup Mode）**——六个强制性问题：

| # | 问题 | 追问目标 | 危险信号 |
|---|------|---------|---------|
| Q1 | 需求真实性 | 具体的行为证据，而非"有人感兴趣" | "500 人注册了候补名单"——这不是需求 |
| Q2 | 现状 | 用户现在怎么解决这个问题，即使很差 | "没有解决方案"——通常意味着问题不够痛 |
| Q3 | 极致具体 | 说出一个真实的人名、职位、痛点 | "企业客户"——你不能给一个类别发邮件 |
| Q4 | 最窄楔子 | 本周能让人掏钱的最小版本 | "需要先建完整平台"——对架构而非价值上瘾 |
| Q5 | 观察与惊讶 | 你看过用户使用产品但不帮忙吗？ | "我们做了调查问卷"——调查问卷会撒谎 |
| Q6 | 未来适配 | 3 年后你的产品更重要还是更不重要？ | "市场年增长 20%"——增长率不是愿景 |

**建设者模式（Builder Mode）**——更轻松、更有创造力：
- 这个东西最酷的版本是什么？
- 你会展示给谁看？什么会让他们说"哇"？
- 到达可用/可分享状态的最短路径是什么？

#### 运作原则（可迁移）

这些原则不仅适用于 AI skill，更是一套通用的产品思维：

1. **具体性是唯一货币。** 模糊的回答会被追问。
2. **兴趣不等于需求。** 行为才算数，金钱才算数。
3. **用户的话比创始人的 pitch 更真实。** 用户描述的价值和你营销文案说的永远有差距。
4. **观察，不要演示。** 引导式演示教会你的东西为零。
5. **现状是你真正的竞争对手。** 不是另一家创业公司——是用户现在凑合用的 spreadsheet + Slack。
6. **早期狭窄胜过宽泛。** 最小版本 → 扩展。

#### 创始人信号追踪

`/office-hours` 会在整个对话中追踪"创始人信号"：

- 能否描述一个真实的问题（非假设）
- 是否提到了具体的人（人名，不是类别）
- 是否在前提挑战时推回（conviction，not compliance）
- 是否展示了品味——在乎细节
- 是否展示了能动性——在建造，不只是在计划

这些信号的数量决定了最终"Garry 的私人话语"的层级——从"你可能不知道自己可以成为创始人"到"gstack 认为你是最优秀的人之一"。

### 5.2 `/plan-ceo-review` — CEO/创始人视角

#### 哲学

> 你不在这里盖章批准计划。你在这里让它变得非凡。

#### 四种模式

```
┌─────────────┬──────────────┬──────────────┬──────────────┬────────────────────┐
│             │  EXPANSION   │  SELECTIVE   │  HOLD SCOPE  │  REDUCTION         │
├─────────────┼──────────────┼──────────────┼──────────────┼────────────────────┤
│ 范围        │ 往上推        │ 保持 + 提供  │ 维持         │ 往下推              │
│ 推荐态度     │ 热情         │ 中立         │ N/A          │ N/A                │
│ 10x 检查    │ 必须         │ 作为可选项   │ 可选         │ 跳过               │
│ 柏拉图理想   │ 是           │ 否           │ 否           │ 否                 │
│ 复杂度问题   │ "够大吗？"   │ "对吗+还有  │ "太复杂了吗？"│ "是最低限度吗？"    │
│             │              │  什么诱人的？"│              │                    │
└─────────────┴──────────────┴──────────────┴──────────────┴────────────────────┘
```

#### 认知模式（可迁移的 CEO 思维）

这些不是检查清单项目，而是**思维本能**——来自 Bezos、Jobs、Munger、Horowitz 等人的决策模式：

1. **分类本能**（Bezos）：每个决策按可逆性 x 影响力分类。大多数是双向门——快速行动。
2. **反转反射**（Munger）：对于每个"如何赢？"也问"什么会让我们失败？"
3. **聚焦即减法**（Jobs）：主要价值在于决定**不做什么**。从 350 个产品砍到 10 个。
4. **速度校准**（Bezos）：70% 信息就够做决策。快速是默认。
5. **代理怀疑**：我们的指标还在服务用户，还是已经变成自我参照的？
6. **时间深度**：以 5-10 年为弧度思考。
7. **意志力即战略**（Altman）：世界向在一个方向上足够长时间足够用力推的人让步。

#### 十个审查部分

1. **架构审查** — ASCII 图表、依赖关系、四条数据路径（正常/nil/空/错误）
2. **错误与救援映射** — 每个可能失败的方法的完整异常表
3. **安全与威胁建模** — 攻击面、输入验证、授权
4. **数据流与交互边界** — 交互边界案例表
5. **代码质量审查** — DRY、命名、复杂度
6. **测试审查** — 完整的测试矩阵图
7. **性能审查** — N+1 查询、内存、缓存
8. **可观测性** — 日志、指标、告警、运行手册
9. **部署与上线** — 迁移安全、特性标志、回滚
10. **长期轨迹** — 技术债务、路径依赖、可逆性

### 5.3 `/plan-eng-review` — 工程经理视角

> 不是"缩小想法"。是**让想法可构建**。

#### 认知模式（工程经理思维）

1. **无聊是默认的**（McKinley）："每个公司大约只有三个创新代币。其他一切都应该是成熟技术。"
2. **渐进优于革命**：绞杀者模式，不是大爆炸。金丝雀，不是全局部署。
3. **为疲惫的 3am 工程师设计**，不是你最好的工程师在最好的一天。
4. **本质 vs 偶发复杂性**（Brooks）：在添加任何东西之前——"这是在解决真实问题还是我们自己创造的问题？"
5. **让变更简单，然后做简单的变更**（Beck）：先重构，再实现。

#### 关键输出

- 强制性 ASCII 图表（数据流、状态机、依赖图）
- 测试计划工件（写入 `~/.gstack/projects/`，`/qa` 自动消费）
- 失败模式注册表（每个代码路径一种现实的失败方式）
- TODOS.md 更新（每个 TODO 都有完整的上下文、原因、优缺点）

### 5.4 `/review` — Staff Engineer 视角

#### 两遍审查

**Pass 1（关键）：**
- SQL 和数据安全（字符串插值、TOCTOU 竞态）
- 竞态条件和并发
- LLM 输出信任边界（AI 生成的值必须验证后才能写入数据库）
- 枚举和值完整性（新枚举值必须追踪到每个消费者）

**Pass 2（信息性）：**
- 条件副作用
- 魔法数字和字符串耦合
- 死代码和一致性
- LLM Prompt 问题（0-索引列表、工具不匹配）
- 测试空缺
- 前端/视图问题

#### Fix-First 哲学

> 每个发现都得到行动——不仅仅是关键的。

- **AUTO-FIX**：机械性修复直接应用
- **ASK**：需要判断的问题批量呈现给用户

#### 范围漂移检测

在审查代码质量之前，先检查：**他们做的是被要求的东西吗——不多也不少？**

### 5.5 `/qa` — QA 主管视角

这是 Garry Tan 说的"massive unlock"——让他从 6 个并行 worker 增加到 12 个。

#### 核心循环

```
发现 bug → 定位源码 → 最小修复 → 原子提交 → 重新测试 → 生成回归测试
```

#### 自我调节（WTF 可能性）

```
WTF-LIKELIHOOD:
  起始 0%
  每次回滚:                +15%
  每个修复触及 >3 文件:     +5%
  修复 15 次之后:           每次额外 +1%
  触及无关文件:             +20%
```

如果 WTF > 20%，立即停止，向用户展示进度。硬性上限：50 次修复。

#### 回归测试生成

每个验证通过的修复都会自动生成回归测试：
1. 研究项目现有测试模式
2. 追踪 bug 的代码路径
3. 测试必须在无修复时失败、有修复时通过
4. 包含完整归因注释

### 5.6 `/ship` — 发布工程师视角

**设计目标：** 用户说 `/ship`，下一个看到的就是 PR URL + 自动同步的文档。

#### 完整步骤

1. **预飞检查** — 分支、状态、Review Readiness Dashboard
2. **合并基础分支** — 在测试之前合并，确保测试运行在合并状态上
3. **运行测试** — 并行运行所有测试套件
4. **Eval 套件**（条件性）— 如果 prompt 文件变更，强制运行评估
5. **测试覆盖审计** — ASCII 覆盖率图表，自动生成缺失测试
6. **Pre-Landing Review** — 应用 review checklist
7. **版本号** — 自动判断 MICRO/PATCH/MINOR/MAJOR
8. **CHANGELOG** — 从 diff 自动生成
9. **TODOS.md** — 自动标记已完成项
10. **可二分提交** — 拆分为逻辑性 commit
11. **验证门** — 如果代码变了，重新跑测试
12. **推送 + 创建 PR**
13. **自动调用 `/document-release`** — 同步所有文档

#### 铁律

> **没有新鲜验证证据就不能声称完成。**
>
> "应该能工作了" → 运行它。
> "我很自信" → 自信不是证据。
> "我之前已经测试过了" → 代码在那之后变了，再测一次。
> "这是微不足道的变更" → 微不足道的变更会搞垮生产。

### 5.7 `/investigate` — 系统化调试

#### 铁律

> **没有根因调查就没有修复。**

四阶段流程：
1. **根因调查** — 收集症状，读代码，检查最近变更，复现
2. **模式分析** — 匹配已知模式（竞态、空值传播、状态损坏...）
3. **假设测试** — 先验证假设再写修复。3 次失败后强制停止
4. **实现** — 修复根因而非症状，写回归测试，运行全套测试

#### 范围锁定

形成假设后，自动锁定编辑到受影响的模块（通过 `/freeze`），防止调试时意外修改无关代码。

### 5.8 `/retro` — 工程回顾

追踪 14 个维度的指标：

- 提交数、贡献者数、PR 数
- 总插入/删除、净 LOC
- 测试 LOC 比率
- 活跃天数、检测到的会话数
- 每会话小时 LOC
- 团队连续发布天数
- AI 辅助提交百分比

**团队意识：** 为每个贡献者生成个人部分，包括具体的赞扬和成长建议——锚定在实际 commit 上，不是泛泛而谈。

### 5.9 `/design-consultation` — 从零构建设计系统

#### SAFE/RISK 框架

这是设计 skill 的精华。不只是推荐"好看的"——而是区分：

- **SAFE 选择**（类别基线）：你的用户期望这些。遵循行业惯例。
- **RISK 选择**（你的产品获得自己的面孔）：刻意偏离惯例，附带清晰的理由。

> "安全选择让你在类别中有文化素养。风险选择是你的产品变得令人难忘的地方。"

#### AI Slop 反模式（永远不推荐）

- 紫色/紫罗兰色渐变作为默认强调色
- 带图标的三列特性网格
- 一切居中、均匀间距
- 统一的圆角
- 渐变按钮作为主要 CTA
- 通用的 stock photo 风格 hero 区域

### 5.10 `/office-hours` 的"一件事"时刻

在设计文档批准后，`/office-hours` 会输出一个精心设计的收尾序列：

**Beat 1：** 信号反思 — 引用用户在对话中说过的原话，展示观察力
**Beat 2：** "One more thing." — 重置注意力
**Beat 3：** Garry 的个人话语 — 根据创始人信号强度分三个层级：
- **顶级**（3+ 强信号）："gstack 认为你是最优秀的人之一"
- **中级**（1-2 信号）："你在建造真实的东西"
- **基础级**（所有人）："你展示的技能——品味、野心、能动性——正是我们在 YC 创始人中寻找的"

## 6. 浏览器引擎：给 AI 一双眼睛

### 6.1 为什么需要浏览器

> Claude Code 说 "I SEE THE ISSUE" 然后真正修复它、生成回归测试、验证修复——这改变了我的工作方式。

传统 AI 编码助手是"盲人"——它们读代码但不能看到渲染后的页面。gstack 的浏览器引擎让 agent 能：

- 导航到 URL
- 截屏
- 读取无障碍树（snapshot）
- 点击元素
- 填写表单
- 检查控制台错误
- 监控网络请求

### 6.2 守护进程模型

不是每个命令启动一个浏览器（3-5 秒延迟 + 丢失状态），而是运行一个长期存活的 Chromium 守护进程：

- **持久状态**：登录一次，保持登录。打开标签页，保持打开
- **亚秒命令**：第一次调用后，每个命令 ~100-200ms
- **自动生命周期**：首次使用时自动启动，空闲 30 分钟后自动关闭
- **版本自动重启**：如果二进制版本不匹配运行中的服务器，自动杀死旧服务器启动新的

### 6.3 安全模型

- 仅绑定 localhost
- Bearer token 认证（UUID，mode 0o600）
- Cookie 在内存中解密，从不以明文写入磁盘
- 浏览器 Cookie 数据库以只读方式打开
- Shell 注入防护（参数数组，非字符串插值）

### 6.4 Handoff 机制

当 AI 被困住（CAPTCHA、认证墙、MFA），`$B handoff` 打开一个可见的 Chrome，保留所有 cookie 和标签页。用户解决问题后，`$B resume` 从中断处继续。

## 7. 安全与治理：让高速不等于失控

### 7.1 三层安全

| Skill | 机制 | 保护范围 |
|-------|------|---------|
| `/careful` | PreToolUse hook 检测危险命令 | rm -rf, DROP TABLE, force-push, git reset --hard, kubectl delete |
| `/freeze` | PreToolUse hook 拦截越界编辑 | 限制 Edit/Write 到指定目录 |
| `/guard` | 同时激活 careful + freeze | 最大安全性（适合生产环境工作） |

### 7.2 安全白名单

`/careful` 不会对这些常见操作发出警告：
- `rm -rf node_modules` / `.next` / `dist` / `__pycache__` / `.cache` / `build`

### 7.3 Review Readiness Dashboard

`/ship` 在发布前检查审查就绪状态——哪些 review 已运行、哪些过时、是否有未解决的问题。这不是可选的——它是发布门控。

## 8. 多模型协作：跨 AI 的第二意见

### 8.1 `/codex` — 多 AI 第二意见

gstack 不局限于单一 AI。`/codex` 调用 OpenAI 的 Codex CLI 获取独立的第二意见：

- **Review 模式**：独立的 diff 审查，有通过/失败门控
- **对抗模式**：主动尝试破坏你的代码（最大推理深度 `xhigh`）
- **咨询模式**：问 Codex 任何问题，支持对话延续

当 `/review`（Claude）和 `/codex`（OpenAI）都审查了同一分支后，生成**跨模型分析**：
- 两者都发现的问题
- 只有 Codex 发现的问题
- 只有 Claude 发现的问题
- 一致率

### 8.2 多 Agent 支持

gstack v0.9.0 开始，skill 不仅在 Claude Code 上工作，还支持 Codex CLI、Gemini CLI 和 Cursor。通过 `./setup --host auto` 自动检测并安装到所有可用的 agent。

## 9. 模板引擎与质量保证

### 9.1 三层测试

| 层级 | 内容 | 成本 | 速度 |
|------|------|------|------|
| Tier 1 — 静态验证 | 解析 SKILL.md 中的每个 `$B` 命令，对照注册表验证 | 免费 | <2s |
| Tier 2 — E2E 测试 | 启动真实 Claude 会话，运行每个 skill，检查错误 | ~$3.85 | ~20min |
| Tier 3 — LLM 评判 | Sonnet 在清晰度/完整性/可操作性上评分 | ~$0.15 | ~30s |

### 9.2 基于 Diff 的测试选择

`test:evals` 基于 `git diff` 自动选择要运行的测试。每个测试声明其文件依赖。全局触控文件（session-runner、eval-store、llm-judge）的变更触发所有测试。

### 9.3 Eval 持久化

测试结果持久化到 `~/.gstack-dev/evals/`：
- **增量保存**：每个测试后写入 `_partial-e2e.json`（原子写入）
- **最终保存**：带时间戳的评估文件
- `eval:compare` 比较两次运行
- `eval:summary` 聚合所有运行的统计

## 10. 并行化策略：10-15 个 Sprint 同时跑

### 10.1 为什么并行能工作

> "没有流程，十个 agent 是十个混乱源。有流程，每个 agent 确切知道该做什么和何时停下。"

Sprint 结构使并行成为可能：
- 一个会话运行 `/office-hours` 探讨新想法
- 另一个在某个 PR 上做 `/review`
- 第三个实现一个功能
- 第四个在 staging 上跑 `/qa`
- 还有六个在其他分支上工作

### 10.2 会话感知

当 3+ gstack 会话同时运行时，ELI16 模式激活——每个问题都重新交代上下文（项目名、分支名、正在做什么），因为用户在切换窗口时很容易搞混。

### 10.3 端口隔离

每个浏览器守护进程使用 10000-60000 之间的随机端口。10 个工作空间可以各自运行自己的浏览器守护进程，零配置，零端口冲突。

## 11. 方法论精华：可迁移的思维方式

### 11.1 核心思维模型

以下是 gstack 中最有价值的、可迁移到任何工程实践中的思维方式：

#### "从痛苦出发，不是从功能出发"

`/office-hours` 的核心做法是：用户说"我要做一个日程简报 app"，它不去实现日程简报 app。它问**痛苦是什么**。然后基于痛苦重新定义产品。

**迁移方式：** 在开始任何项目之前，问"你描述的是一个功能还是一个问题？如果是功能，背后的问题是什么？"

#### "强制性的替代方案"

gstack 在 `/office-hours` 和 `/plan-ceo-review` 中都强制要求生成 2-3 个替代方案。这不是可选的。

**为什么有效：** 第一个方案几乎总是"最显而易见的"，而非最好的。强制生成替代方案迫使你思考不同的架构、不同的权衡、不同的起点。

#### "图表暴露隐藏假设"

`/plan-eng-review` 大量使用 ASCII 图表——数据流、状态机、依赖图、错误路径。

> "当你强迫 LLM 画出系统时，它们变得完整得多。图表让含糊的规划变得困难得多。"

#### "四条数据路径"

每个数据流都有四条路径：

```
正常路径 (数据正确流过)
Nil 路径 (输入为 nil/缺失)
空路径 (输入存在但为空/零长度)
错误路径 (上游调用失败)
```

**迁移方式：** 为每个新的数据流追踪这四条路径。这一个习惯就能消除大量生产 bug。

#### "沉默失败是关键缺陷"

> 每个失败模式都必须可见——对系统、对团队、对用户。如果一个失败可以悄无声息地发生，那就是计划中的关键缺陷。

#### "Fix-First，不是 Read-Only"

传统代码审查是"发现问题 → 写评论 → 等作者修"。gstack 的做法是"发现问题 → 如果是机械性的就直接修 → 只有需要判断的才问"。

#### "验证门"

"我很自信"不是证据。在 push 之前：
- 如果代码变了 → 重新跑测试
- 如果有构建步骤 → 运行它
- 贴出新鲜的输出

### 11.2 错误哲学

gstack 的错误信息是为 AI agent 设计的，不是为人类。每条错误消息必须是可操作的：

- "Element not found" → "Element not found or not interactable. Run `snapshot -i` to see available elements."
- 超时 → "Navigation timed out after 30s. The page may be slow or the URL may be wrong."

**原则：agent 读到错误后应该知道下一步做什么，无需人类干预。**

### 11.3 承诺模式

gstack 的 CLAUDE.md 要求**总是二分提交**（bisect commits）。每个 commit 是一个逻辑变更。好的二分示例：
- 重命名/移动 与 行为变更 分开
- 测试基础设施 与 测试实现 分开
- 模板变更 与 生成文件再生成 分开
- 机械重构 与 新功能 分开

## 12. 实践指南：如何学习和借鉴 gstack

### 12.1 渐进式学习路径

**第一阶段：体验（30 分钟）**
1. 安装 gstack
2. 运行 `/office-hours` — 描述你正在构建的东西
3. 体验"重新定义问题"的过程

**第二阶段：理解 Sprint 流程（2-3 小时）**
1. 在任何有改动的分支上运行 `/review`
2. 运行 `/qa` 对你的 staging URL
3. 运行 `/ship` 完成一次完整发布
4. 运行 `/retro` 查看你的工程指标

**第三阶段：深入方法论（半天）**
1. 阅读 `/plan-ceo-review` 的"认知模式"部分
2. 阅读 `/plan-eng-review` 的"工程经理思维"部分
3. 阅读 `/investigate` 的铁律和四阶段流程
4. 理解 Review Readiness Dashboard 如何连接所有 skill

**第四阶段：定制和扩展（持续）**
1. Fork gstack，开启 Contributor Mode
2. 修改 `.tmpl` 文件，定制适合你团队的流程
3. 参考 `gen-skill-docs.ts` 理解模板系统
4. 编写你自己的 skill

### 12.2 最有价值的可借鉴元素

如果你不使用 gstack 本身，以下元素仍然可以独立借鉴：

1. **强制性替代方案生成** — 任何重要决策前都生成 2-3 个替代方案
2. **四条数据路径** — 为每个数据流追踪正常/nil/空/错误路径
3. **铁律：验证优于自信** — 永远不要声称"完成"而没有实际运行测试
4. **Fix-First Review** — 能直接修的就直接修，只问真正需要判断的
5. **WTF 可能性计算** — 自动化的"该停下了"机制
6. **CEO 认知模式** — 可逆性分类、反转反射、聚焦即减法
7. **ELI16 会话感知** — 多任务并行时的上下文重建
8. **完备性原则** — AI 时代，完整实现几乎不比偷懒贵

### 12.3 gstack 的核心启示

> "这个东西的威力其实是思维方式，而不是 skill。他是把一种非常独特的思维方式用 skill 的形式展现出来了。"

这正是 gstack 真正的价值所在。它不是一个工具集——它是一个**编码了世界顶级创业导师和工程管理者思维方式的系统**。每个 SKILL.md 文件都包含：

- 一种**角色认同**（你是 CEO / 工程经理 / QA 主管...）
- 一套**认知模式**（如何思考这类问题）
- 一组**强制约束**（铁律、停止条件、验证门）
- 一个**结构化流程**（步骤、输入、输出、交接）

当这些元素被组合在一起，交给一个足够强大的 AI 模型，并提供足够密度的项目上下文时——你得到的不是一个聊天机器人的回答，而是一份**真正有洞察力的战略规划**。

这就是为什么那位 UP 主说"仅仅是执行了 `/office-hours` 就得到了一份目标年入 5000 万的战略规划报告"——因为 `/office-hours` 不是在问你"你想做什么功能"，它是在用 YC 合伙人的思维模式审视你的业务，追问你的假设，挑战你的前提，然后用强制性替代方案展示你可能没考虑过的路径。
